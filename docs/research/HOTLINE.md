## 1. XML — повна специфікація

### 1.1. Загальні елементи

| Елемент | Обов'язк. | Тип / обмеження |
|---|---|---|
| `<?xml ?>` | так | Перший рядок з нульового символу. `encoding` — **UTF-8** або **windows-1251** |
| `<price>` | так | Кореневий елемент |
| `<date>` | ні | Формат строго `РРРР-ММ-ДД ГГ:ХХ` (`2026-07-13 17:00`) |
| `<firmName>` | ні | Назва магазину |
| `<firmId>` | ні | ID магазину з акаунта Hotline |
| `<rate>` | умовно | Курс USD. **Обов'язковий, якщо ціни в USD.** Нам не потрібен — ми віддаємо `priceRUAH` |
| `<categories>` | **так** | Список категорій магазину |
| `<delivery>` | ні | Способи доставки (рівень магазину), атрибути — див. 1.4 |
| `<store>` | ні | Пункти видачі |
| `<items>` / `<item>` | **так** | Товари |

### 1.2. `<categories>`

```xml
<category>
  <id>ціле число, ≤18 знаків, унікальне і НЕЗМІННЕ між вивантаженнями</id>
  <parentId>id батьківської (опційно — задає ієрархію)</parentId>
  <name>назва категорії</name>
</category>
```

### 1.3. `<item>` — кожен тег

| Тег | Обов'язк. | Тип / обмеження |
|---|---|---|
| `<id>` | **ТАК** | ≤20 символів; тільки `0-9 a-z A-Z - _`. Унікальний і **незмінний**. **Заборонено перевикористовувати id видаленого товару для нового.** |
| `<categoryId>` | **ТАК** | Один товар = рівно **одна** категорія. Кінцева категорія має відповідати рубрикатору Hotline |
| `<vendor>` | **ТАК** | Рівно один виробник. **Заборонено вказувати країну-виробника** |
| `<name>` | **ТАК** | Тільки назва моделі, одна. Без реклами, без переліку, без описової інформації |
| `<url>` | **ТАК** | Тільки **HTTPS**. Тільки україномовна сторінка того домену, що в анкеті |
| `<priceRUAH>` | **ТАК** | Ціна в грн з усіма податками. **Заборонено роздільники розрядів** (`15400`, не `15 400`) |
| `<code>` | умовно | **Тільки артикул від виробника (MPN).** Явна заборона: внутрішні коди магазину, назви товарів тощо |
| `<barcode>` | умовно | Тільки штрихкод виробника. Заборонено класти будь-що інше |
| `<priceRUSD>` | ні | Ціна в USD; у картках показуються тільки грн |
| `<description>` | ні | **На сайті не відображається.** Використовується для формування характеристик. **HTML заборонено.** Плоский текст виду `Ключ: значення; Ключ: значення;` |
| `<image>` | ні | Див. 1.6 |
| `<stock>` | ні | `В наявності` / `Під замовлення` / `Немає` — див. §6 |
| `<shipping>` | **так для "Під замовлення"** | Ціле або діапазон днів; атрибути `workdays`/`saturday`/`sunday` — див. §6 |
| `<guarantee>` | ні | Атрибути `type`, `unit` — див. §5 |
| `<param name="...">` | ні | Технічні характеристики. Назви/одиниці/значення мають збігатися зі структурою характеристик каталогу Hotline |
| `<param name="Країна виготовлення">` | ні | Окремо задокументований параметр |
| `<region>` | ні | Обласний центр українською, через кому. Якщо не вказано — показ у всіх областях. Фід має пріоритет над налаштуванням в акаунті |
| `<delivery>` | ні | Доставка на рівні товару (`cost`, `freeFrom`, `unavailable`) |
| `<payment>` | ні | Оплата частинами і комісії — див. §7 |
| `<condition>` | ні | `0` новий (дефолт), `1` refurbished (Grade A), `2` уцінений, `3` б/в. **Товари з `3` на сайт не вивантажуються** |
| `<custom>` | ні | Ціле; для відбору в Аукціоні Hotline. Вимагає наявності `<id>` |

### 1.4. `<delivery>` (окрема сторінка специфікації)

| Атрибут | Обов'язк. | Значення |
|---|---|---|
| `id` | так | ціле |
| `type` | так | `pickup` \| `warehouse` \| `pochtomat` \| `address` |
| `carrier` | так для `warehouse`/`pochtomat`/`address` | **тільки `SLF`, `ME`, `NP`, `UP`** |
| `cost` | ні | грн; `0` = безкоштовно; `null` = «за тарифами перевізника» |
| `freeFrom` | ні | сума замовлення, від якої безкоштовно |
| `city` | ні | тільки для `type="address"`, українською, через кому |

**Пастка для Complex:** перевізник **Delivery не входить в enum** (`SLF/ME/NP/UP`). Meest = `ME`, Нова Пошта = `NP`, Укрпошта = `UP`. «Delivery» у фід не передаємо (або мапимо на `SLF` — власна доставка). Самовивіз в Ужгороді = `type="pickup"` + `<store>`.

### 1.5. Екранування

`<` → `&lt;`, `>` → `&gt;`, `&` → `&amp;`, `'` → `&apos;`, `"` → `&quot;`.
Заборонено: ASCII 0–31 (крім 9, 10, 13); **HTML-розмітка в усіх без винятку елементах**.

### 1.6. Зображення

- Формати: **JPEG (бажано)**, GIF/PNG **без прозорих областей**.
- **Шлях до файлу — тільки латиниця, цифри, `-`, `_`.**
- Білий фон; без написів, логотипів бренду, текстів, водяних знаків; без заглушок «немає фото».
- Розмір у пікселях у специфікації **не задано**.

> **Наслідок для R2:** ключі об'єктів у Cloudflare R2 треба генерувати транслітом/slug'ом (`siemens-hz66d910-1.jpg`), **не** кирилицею і не з пробілами. Це вимога фіда, а не побажання.

Розпарсено `example_ukr (1).xls` (xlrd; лист `Лист1`, 9 рядків × 14 колонок; `Лист2`/`Лист3` порожні):

| # | Заголовок | Аналог у XML | Типи в шаблоні |
|---|---|---|---|
| A | Категорія товару | `<category><name>` | text |

| C | Назва товару | `<name>` | text |
| D | Код товару (артикул від виробника) | `<code>` | text/number/порожньо |
| E | id товару | `<id>` | number |
| F | Штрихкод | `<barcode>` | number/порожньо |
| G | Опис товару | `<description>` | text/порожньо |
| H | Ціна роздр., грн. | `<priceRUAH>` | number |
| I | Гарантія | `<guarantee>` | **змішано**: `12`, `36`, `12 місяців від магазину`, `14 днів від магазину`, `від виробника`, порожньо |

**Чого в XLS немає (порівняно з XML):** `param` (характеристики), `payment` (оплата частинами і комісії), `delivery`, `store`, `region`, `custom`, `priceRUSD`/`rate`, `parentId` (ієрархія категорій — тільки плоска назва рядком).

## 3. XML чи XLS — обґрунтування

**Однозначно XML.** Гіпотеза підтверджена, і причини сильніші, ніж «простіше генерувати»:

1. **`<param>` є тільки в XML.** Правила Hotline: за наявності **унікальних пропозицій** магазин **зобов'язаний** передати характеристики — через шаблон в акаунті або через `param` у фіді. У Complex є повна таблиця характеристик (трійки з колонки J). У XLS передати їх нічим — залишиться тільки ручне заповнення шаблонів в кабінеті.
2. **Оплата частинами (`<payment>`) є тільки в XML.** Пункт §7 ТЗ у XLS не реалізується взагалі.
3. **Гарантія в XLS — неструктурована.** Одна колонка змішує число і вільний текст, тип гарантії (виробник/магазин) не відділений. У XML це атрибути `type`/`unit`.
4. **Ієрархія категорій.** XML дає `parentId`; XLS — плоский рядок.
5. **Доставка і пункти видачі** (`delivery`, `store`) — тільки XML.
6. **XLS — формат другого класу** у самої Hotline: офіційно не підходить для окремих розділів каталогу.

**Підводні камені XML, які треба врахувати (не блокери):**
- `<image>` у специфікації описаний як **одне** зображення на товар — множинна галерея у фід не передається в жодному форматі.
- `<description>` **не відображається на сайті Hotline** і **не приймає HTML**. Наш rich-HTML опис туди класти не можна — тільки плоскі `Ключ: значення;` або взагалі опустити на користь `param`.
- `<code>` — **тільки MPN**. Класти туди наш внутрішній артикул (`2400042`) **прямо заборонено**. Це знімає невизначеність з відкритого питання №2 в `INPUTS.md`: за відсутності MPN тег просто **не віддаємо**; внутрішній артикул йде в `<id>`.
- Ціни — **без роздільників розрядів**. Наш парсер бере `6 600,00` → у фід має піти `6600`.

## 4. Як Hotline забирає фід

- Магазин **вказує в акаунті URL** на опублікований у Інтернеті файл. Там же задається **розклад** автозавантаження. Є кнопка **«Оновити»** для термінового забору.
- Протокол: **HTTP(S)**.
- **Авторизація заборонена.** JavaScript для перенаправлення або динамічного підвантаження — заборонений. Посилання на HTML-сторінку не обробляється. **Блокування доступу робота неприпустиме.**
- Оновлювати можна **кілька разів на добу**.
- **Фід публікується 7 днів.** Якщо за цей час не завантажено оновлений фід — публікація **припиняється**. Тобто наш Celery-beat має оновлювати фід мінімум раз на добу (реально — 2–4 рази).
- Нові товари, яких ще немає в каталозі Hotline, додаються модераторами **в робочий час**.

### Вимоги до Caddy (критично)

Hotline **не публікує** ані User-Agent, ані IP-діапазони свого робота — тому **allowlist за UA/IP будувати не можна**. Єдина надійна стратегія — зробити шлях фіда повністю «тупим»:

- Віддавати за **канонічним постійним URL** (напр. `https://complex.example/feeds/hotline.xml`) з **HTTP 200 напряму**, без `301`/`302` (у т.ч. без редіректу `www` → apex, без редіректу на локаль `/ua/`).
- **Жодного Cloudflare/Caddy challenge, rate-limit, basic-auth, WAF-правила, `robots.txt`-блоку** на цьому шляху.
- `Content-Type: application/xml; charset=utf-8`.
- Не віддавати HTML-сторінку помилки зі статусом 200.
- Обов'язково підтримувати HTTPS з валідним сертифікатом (`<url>` товарів теж мусить бути HTTPS).
- Рекомендація: генерувати фід у файл (Celery) і віддавати статикою через `file_server`, а не рендерити на льоту — це прибирає таймаути на 150k товарів.

## 5. Гарантія

```xml
<guarantee type="manufacturer">12</guarantee>
```

- Значення — **число, за замовчуванням у місяцях**.
- `unit="days"` — якщо термін у днях.
- `type="manufacturer"` — офіційна гарантія виробника; `type="shop"` — гарантія магазину.
- **Для одного товару можна вказати лише один тип.**

**Мапінг для Complex:** замовник підтвердив «Гарантія від виробника» → завжди `type="manufacturer"`. Термін парсимо з характеристики `Гарантійний термін (міс): 12` → `<guarantee type="manufacturer">12</guarantee>`. Якщо характеристики немає — тег **опускаємо** (він опційний), не підставляємо дефолт.

## 6. Доступність (`<stock>`)

Рівно три допустимі значення (регістр і формулювання як у специфікації):

| Значення | Поведінка Hotline |
|---|---|
| `В наявності` | Вивантажується |
| `Під замовлення` | Вивантажується (якщо не вимкнено фільтром менеджера). **Обов'язково потрібен `<shipping>`** |
| `Немає` | **На сайт не вивантажується** |

- На сторінці товару значення `stock` **не показується** — воно керує фільтром завантаження.
- Будь-яке інше значення = порожньому/відсутньому тегу.

### `<shipping>` — «Відвантаження зі складу»

Ціле число або діапазон днів. Градації показу: `0` → «сьогодні»; `1–2` → «1-2 дні»; `3–6` → «3-6 днів»; `7–14` → «7-14 днів»; `>15` → «більше 14 днів». За діапазону значущим є максимум. Якщо тега немає — Hotline пише «уточнюйте у продавця».

Час відсічки для «сьогодні»:
```xml
<shipping workdays="16:00" sunday="14:00" saturday="18:00">0</shipping>
```

**Мапінг для Complex:** `qty > 0` → `В наявності` + `<shipping>0..1</shipping>`; товар є в прайсі, але `qty = 0` → `Під замовлення` + `<shipping>` з терміну постачання; товару немає в поточному зрізі прайсу → `Немає`.

## 7. Оплата частинами

Так, тег є — **тільки в XML**:

```xml
<payment type="installment"
         from="Монобанк"
         min_payment="500"
         max_period="24"
         no_down_payment="true">true</payment>
```

| Атрибут | Значення |
|---|---|
| `from` | Банк або `Магазин` (якщо послугу надає сам торговець). Допустимі: **Магазин, Monobank, Privatbank, Sense bank, A-bank, PUMB, Oschadbank, OTPBank, Izibank, Ukrsibbank** |
| `min_payment` | Мінімальна сума платежу |
| `max_period` | Максимальна кількість місяців |
| `no_down_payment` | `true`/`false` — відсутність першого внеску |
| (тіло тега) | `true` / `false` |

### Комісії за способами оплати (той самий тег, атрибут `fee` = відсоток)

`pay-card` (Visa/MC), `card-card` (карта→карта), `vat` (безготівка з ПДВ), `non-vat` (безготівка без ПДВ), `card-in-shop` (карта в пункті видачі), `cash-on-delivery` (післяплата).

```xml
<payment type="pay-card" fee="0">true</payment>
<payment type="cash-on-delivery" fee="2">true</payment>
```

**Для Complex:** розстрочка йде через LiqPay → у `from` вказуємо конкретні банки-партнери, які реально доступні в чекауті. Потрібне уточнення в замовника: список банків + `min_payment` + `max_period`. Поки не підтверджено — тег не віддаємо (він опційний).

## 8. Категорії та рубрикатор

**Механіка:** у фіді ми віддаємо **власні** категорії з **власними** `<id>`. Hotline не має числових ID для мапінгу. Прив'язка робиться так:

> «Кінцева категорія, до якої віднесено товар у товарному фіді, **повинна відповідати аналогічній в каталозі Хотлайн**».

Тобто ключ зіставлення — **назва кінцевої (листової) категорії**, а не ID.

**Довідник є і завантажується:**
`https://hotline.ua/download/hotline/hotline_tree_uk.csv`

Характеристики файлу (перевірено):
- HTTP 200, `application/octet-stream`, ~28 КБ, **1223 рядки**.
- Кодування — **windows-1251** (не UTF-8! потрібен `iconv`/`decode('cp1251')`).
- Формат — не справжній CSV, а **дерево з відступами через `;`**: кількість провідних `;` = глибина.
- Глибина: 4 рівні (17 кореневих розділів → 108 → 663 → 435).

```
Побутова техніка
;Велика побутова техніка
;;Холодильники
;;Пральні машини
;;Варильні поверхні
;;Духовки
;;Кухонні плити
;;Витяжки
;Вбудована техніка
```
(Телевізори — `Побутова техніка ; Телевізори ; Телевізори`; пилососи/роботи-пилососи — рівень 3.)

**Що робимо:**
1. Celery-таск раз на тиждень тягне `hotline_tree_uk.csv`, декодує cp1251, парсить відступи в дерево, зберігає в таблицю `HotlineCategory (path, name, depth, parent)`.
2. У нашій моделі `Category` — поле `hotline_category` (FK на `HotlineCategory`, nullable), обирається в Django Admin автокомплітом.
3. Генератор фіда віддає в `<category><name>` **назву з рубрикатора Hotline**, а не нашу маркетингову назву. `parentId` будуємо з дерева рубрикатора.
4. Товари з незамапленою категорією — **не потрапляють у фід** + WARN у лог (інакше Hotline не прив'яже їх до карток).

## 9. Обмеження (зведено)

| Параметр | Значення |
|---|---|
| Кількість товарів | **≤ 150 000** |
| Розмір у МБ | не регламентовано |
| Кодування | **UTF-8** або **windows-1251** |
| Частота оновлення | кілька разів на добу; **мінімум раз на 7 днів**, інакше публікація зупиняється |
| Формат дати | `РРРР-ММ-ДД ГГ:ХХ` |
| Ціни | без роздільників розрядів |
| HTML-розмітка | **заборонена в усіх елементах** |
| Символи | заборонені ASCII 0–31, крім 9, 10, 13 |
| `<id>` товару | ≤20 симв., `0-9 a-z A-Z - _`, унікальний, незмінний, **без перевикористання** |
| `<id>` категорії | ціле, ≤18 знаків, унікальне і незмінне |
| `<url>` | тільки HTTPS, тільки україномовна сторінка домену з анкети |
| Зображення | JPEG (бажано) / GIF / PNG **без прозорості**; білий фон; без написів/логотипів/водяних знаків; **шлях лише латиниця, цифри, `-`, `_`** |
| Excel | фід на першому листі; без формул, прихованих рядків і колонок |
| Мова контенту | державна (українська) |

## 10. Чек-лист для генератора (що змінити в моделях)

1. `Product.mpn` (nullable) — окреме поле під `<code>`. **Внутрішній артикул туди класти заборонено** — він іде в `<id>`. Закриває відкрите питання №2 в `INPUTS.md`.
2. `Product.barcode` (nullable) — під `<barcode>`.
3. `Category.hotline_category` → FK на нову модель `HotlineCategory` (імпорт з cp1251-CSV).
4. R2-ключі зображень — **slug лише з `[a-z0-9-_]`**, транслітерація обов'язкова.
5. Опис для Hotline ≠ опис для сайту: `<description>` — плоский текст без HTML, або не віддаємо взагалі.
6. `<param>` будуємо з трійок таблиці характеристик, з конкатенацією одиниці до **значення** (`Висота` → `1635 мм`) — це збігається з правилом рендеру з `INPUTS.md` §2.
7. Caddy: окремий route на `/feeds/hotline.xml` — `200`, без редіректів, без auth, без challenge, `file_server` зі згенерованого файлу.
