## 1. Що саме перекладаємо і як воно розпадається на «види»

`TranslationKind` — це не косметика, це визначає модель, батч, валідатор і політику схвалення:

| kind | Джерело | К-сть | Особливість |
|---|---|---|---|
| `attribute_name` | `Attribute.name` | ~600 | **СЛОВНИК.** Один раз. |
| `attribute_value` | `AttributeOption.value` | ~6 000 | **СЛОВНИК.** Один раз. ~35% не перекладаються (числа, бренди) |
| `unit` | `Unit.name` | ~25 | **СЛОВНИК.** `міс→мес`, `год→ч`, `об/хв→об/мин` |
| `category_name` | `Category.name` | ~300 | Словник, впливає на меню + URL |
| `product_name` | `Product.name` | 10 000 | Перекладається **тільки описова частина** (§3) |
| `product_description` | rich HTML | ~6 000 | Найдорожче. Сегментний переклад (§4) |
| `seo_title` / `seo_description` | Product, Category | ~20 000 | Короткі, батчаться по 50 |
| `page_html` / `news_html` | StaticPage, News | ~60 | Той самий пайплайн, що й описи |

**Критично, зі скрінів (INPUTS §2):** одиниця виміру лежить **в окремій колонці** й клеїться до ЗНАЧЕННЯ при рендері. Отже `Unit` — це теж нормалізована таблиця, отже теж словник. Це не «дрібниця»: без цього `12 міс` поїде на фронт як `12 міс` у російській версії.

## 2. КЛЮЧОВА ІДЕЯ: перекладаємо СЛОВНИК, а не товари

### 2.1. Чому це не оптимізація, а єдино правильна архітектура

У наївному підході («візьми товар, перекладі його блок характеристик») рядок `Колір виробу → Чорний` йде в модель **окремо для кожного товару, де він зустрічається**. Наслідки:

1. **Неконсистентність.** Один і той самий термін у різних запитах отримає різний переклад: `Чорний` → `Черный` / `Чёрный`; `Тип управління` → `Тип управления` / `Тип управлением`; `Сенсорне` → `Сенсорное` / `Сенсорный`. Це не гіпотеза — це базова властивість LLM. **Фасетний фільтр після цього розпадається**: два різні рядки = два різні значення фасета = дубльовані чекбокси у фільтрі. Це прямий баг у продакшні.
2. **Витрати ×130.**
3. **Повторний реімпорт прайсу = повторна оплата всього.**

У БД (`DATA_MODEL.md`) `Attribute` та `AttributeOption` — **уже нормалізовані таблиці**. Тобто словник у нас є безкоштовно. Треба просто не ігнорувати його.

### 2.2. Розрахунок для 10 000 товарів × 30 характеристик

Припущення (задокументовані, перевіряються через `client.messages.count_tokens`):
- кирилиця, токенізатор Sonnet 5/Opus 4.7+ (він дає **~30% більше токенів**, ніж старий — це офіційно в доках) → **≈ 2,5 символи/токен**;
- пара `назва: значення` ≈ 45 символів ≈ 18 токенів;
- унікальних `attribute_name` ≈ 600, `attribute_value` ≈ 6 000, `unit` ≈ 25;
- з них реально перекладних (не чисті числа / не бренди) ≈ **4 300**.

| | Наївно (в кожному товарі) | Словник (один раз) |
|---|---|---|
| Одиниць перекладу | 10 000 × 30 = **300 000** | **4 300** |
| Токенів input | 10 000 × 600 = **6,0 M** | 86 запитів × 500 = **0,043 M** |
| Токенів output | **6,0 M** | **0,043 M** |
| **Разом токенів** | **12,0 M** | **0,086 M** |
| Вартість, Sonnet 5 + Batch | 6×$1 + 6×$5 = **$36** | **$0,26** |
| Вартість, Opus 4.8 + Batch | 6×$2,50 + 6×$12,50 = **$90** | **$0,65** |
| Консистентність | ❌ не гарантована | ✅ гарантована структурно |
| Повторний реімпорт | **знову $36** | **≈ $0** (лише нові значення) |

**Економія: ~140× за токенами, ~55–140× за грошима.** Але головний виграш — у тому, що `Чорний` фізично не може перекластися двома способами: він лежить в одному рядку БД.

**Бонус:** словник — це ще й **глосарій для промпту** (§8). Схвалені `attribute_value` йдуть у system-блок як обов'язкова термінологія, і описи товарів починають перекладатися тими самими термінами, що й характеристики. Тобто словник платить двічі.

## 3. Назви товарів: бренд і модель НЕ чіпаємо

`Варильна поверхня газова Bosch PNK6B2P40R` → `Варочная поверхность газовая Bosch PNK6B2P40R`

**Не покладаємось на промпт. Три рівні захисту:**

**Рівень 1 — детерміністичний препроцесинг (головний).**
У нас у БД **вже є** `Product.brand` (FK, з колонки «Виробник» прайсу) та артикул. Отже:

```python
def split_name(name: str, brand: str) -> tuple[str, str]:
    """'Варильна поверхня газова Bosch PNK6B2P40R' -> ('Варильна поверхня газова', 'Bosch PNK6B2P40R')"""
    tail = []
    tokens = name.split()
    # відрізаємо з кінця: бренд + токени, що містять і латиницю, і цифру (= код моделі)
    while tokens and (
        tokens[-1].casefold() == brand.casefold()
        or MODEL_CODE_RE.fullmatch(tokens[-1])   # ^(?=.*[A-Za-z])(?=.*\d)[A-Za-z0-9\-/.]+$
    ):
        tail.insert(0, tokens.pop())
    return " ".join(tokens), " ".join(tail)
```

У Claude їде **тільки** `Варильна поверхня газова`. Бренд і модель фізично не потрапляють у запит → їх неможливо зіпсувати. Збірка назад: `f"{ru_head} {tail}".strip()`.

**Рівень 2 — правило в промпті** (для випадків, коли бренд у назві не знайшовся, напр. `Запчастина для духовки TEKA 83340602`): «будь-який токен, що містить одночасно латинські літери та цифри — це код моделі, копіюй символ-у-символ».

**Рівень 3 — пост-валідація (жорстка, не порада).**

```python
LATIN_TOKEN = re.compile(r"[A-Za-z0-9][A-Za-z0-9\-/.]*")

def validate_preserved(src: str, dst: str) -> bool:
    return Counter(LATIN_TOKEN.findall(src)) == Counter(LATIN_TOKEN.findall(dst))
```

Не збіглося → `status=FAILED`, один ретрай на Opus 4.8, потім у чергу «Потребує уваги». **Жоден невалідований переклад не потрапляє навіть у чергу на схвалення.**

## 4. Описи: HTML із зображеннями, не ламаючи розмітку

Скрін `photo_5372857028075790171` (робот-пилосос): заголовки, абзаци, **фото між абзацами**, посилання. Віддавати такий HTML у модель «як є» — це лотерея: модель може змінити атрибут, «покращити» верстку, зламати `<img src="https://r2...">`.

### Рішення: сегментний переклад через DOM (розмітка не проходить через модель узагалі)

```
HTML → lxml/BeautifulSoup
  ↓ обхід дерева
  для кожного БЛОКОВОГО елемента (p, li, h1..h6, td, figcaption, blockquote):
      inner_html → кодуємо інлайнові теги (<b>, <i>, <a>, <span>) як <0>…</0>, <1>…</1>
      + окремі сегменти для перекладних атрибутів: img@alt, img@title, a@title
  ↓
JSON: [{"id":"s0","uk":"Потужне всмоктування <0>2700 Па</0> прибирає навіть шерсть."}, ...]
  ↓ Claude (structured output, той самий масив, той самий порядок)
JSON: [{"id":"s0","ru":"Мощное всасывание <0>2700 Па</0> убирает даже шерсть."}, ...]
  ↓ підстановка назад у ТОЙ САМИЙ DOM → serialize → bleach (той самий allowlist, що в редакторі)
```

**Що це гарантує by construction:**
- `<img src>`, R2-URL, `class`, `style`, порядок вузлів, вкладеність — **біт-у-біт ідентичні**, бо DOM ми не переписуємо, а лише замінюємо текстові вузли;
- зображення між абзацами лишаються рівно там, де були;
- модель фізично не бачить URL → не може їх «поправити».

**Валідатори (усі три обов'язкові):**

```python
def validate_html_translation(src_html: str, dst_html: str, segments_in, segments_out) -> list[str]:
    errs = []
    if len(segments_in) != len(segments_out): errs.append("segment_count")
    for a, b in zip(segments_in, segments_out):
        if a["id"] != b["id"]: errs.append("id_mismatch")
        if set(PLACEHOLDER_RE.findall(a["uk"])) != set(PLACEHOLDER_RE.findall(b["ru"])):
            errs.append(f"inline_tags:{a['id']}")          # <0>…</0> набір не збігся
        if not validate_preserved(a["uk"], b["ru"]):
            errs.append(f"latin_tokens:{a['id']}")          # бренди/моделі/числа
    if tag_multiset(src_html) != tag_multiset(dst_html): errs.append("tag_multiset")
    if urls(src_html) != urls(dst_html): errs.append("urls")
    return errs
```

Плюс перевірка чисел+одиниць: мультимножина `\d+[.,]?\d*\s*(мм|см|кг|л|Вт|Па|дБ|°C|міс)` у джерелі й перекладі має збігатися (з урахуванням мапи одиниць `міс→мес`).

Помилка валідації → **не в чергу, а у FAILED** + ретрай на Opus 4.8 + алерт.

## 5. Батчинг, кешування, зміна джерела

### 5.1. Структура промпту під кеш (порядок рендеру: `tools → system → messages`)

```python
system=[
    {"type": "text", "text": TRANSLATOR_RULES},            # статичний, ~900 tok
    {"type": "text",
     "text": glossary_block,                               # детермінований, sorted(); ~2 000–8 000 tok
     "cache_control": {"type": "ephemeral", "ttl": "1h"}},  # breakpoint на ОСТАННЬОМУ system-блеці
],
messages=[{"role": "user", "content": json.dumps(payload, ensure_ascii=False, sort_keys=True)}],
```

**Правила, які легко порушити й мовчки втратити кеш** (кеш = **префіксний збіг**, будь-який зміщений байт інвалідує все далі):
- ❌ ніяких `datetime.now()`, лічильників, `uuid` у system;
- ❌ `json.dumps(dict)` без `sort_keys=True`, ітерація по `set()`;
- ❌ глосарій, зібраний у порядку `Model.objects.all()` без `.order_by('pk')`;
- ✅ глосарій **версіонується** (`glossary_version`, регенерація раз на добу вночі), а не оновлюється при кожному схваленні терміна.

**Мінімальний кешований префікс (ВИПРАВЛЕНО, див. §7.6): Opus 4.8 — 4 096 токенів, Haiku 4.5 — 4 096, Fable 5 — 2 048. Для Sonnet 5 офіційне значення не підтверджене — плануємо консервативно на 2 048+.**
Наслідок: system-блок (правила ~900 tok + глосарій) **мусить у сумі перевищувати 4 096 токенів**, бо ретраї та словник ідуть на Opus 4.8. Глосарій на 2 000 токенів цього НЕ перекриває → кеш на Opus 4.8 мовчки не запишеться. Прийоми:
- або віддавати в system **повний** глосарій (усі схвалені терміни, ~6 000–8 000 tok) — тоді поріг перекрито із запасом;
- або перед запуском заміряти `count_tokens` на реальному system-блоці й, якщо < 4 096, або добити його (більше термінів/прикладів), або просто не ставити `cache_control` (кеш усе одно не спрацює, а `cache_control` без ефекту — це просто нуль).

Перевірка: `usage.cache_read_input_tokens > 0`; якщо нуль на повторних запитах — шукай «тихий інвалідатор» зі списку вище **або** префікс коротший за мінімум (у цьому разі помилки не буде — просто `cache_creation_input_tokens = 0`).

**TTL:** для Batch-прогону обов'язково `ttl: "1h"` — при 5-хвилинному TTL кеш встигне протухнути між частинами великого батчу. Перед сабмітом батчу — прогрів одним запитом із `max_tokens: 0` (він виконує prefill і пише кеш, коштує тільки cache-write).

### 5.2. Batch API

- **50% знижки на input і output**, знижки **стакаються з кешуванням** (cache read у батчі = `base × 0.1 × 0.5`).
- До 100 000 запитів / 256 MB на батч; типово < 1 год, максимум 24 год.
- `custom_id = f"te-{entry.id}"`. **Результати приходять у довільному порядку** — ключувати тільки за `custom_id`, ніколи за індексом.
- Sync API (`messages.create`) — тільки для кнопки «Перекласти зараз» в адмінці. Обсяг мізерний, ×2 ціна не болить.

### 5.3. Розміри батчів (один API-запит)

| kind | Сегментів на запит | Чому |
|---|---|---|
| `attribute_name`, `attribute_value`, `unit` | 50 | Короткі; більше — модель починає «зливати» контекст між ними |
| `product_name` | 25 | |
| `seo_*` | 25 | |
| `product_description` | **1 товар = 1 запит** (усі його сегменти) | Потрібен контекст усього опису для узгодженості |

### 5.4. Хеш джерела й обробка змін

```python
def source_hash(text: str) -> str:
    norm = unicodedata.normalize("NFC", text)
    norm = re.sub(r"\s+", " ", norm).strip()     # косметичні зміни пробілів НЕ тригерять переклад
    return hashlib.sha256(norm.encode()).hexdigest()
```

**Пайплайн при синхронізації (`SYNC.md`):**

```
для кожного перекладного поля:
    new_hash = source_hash(new_source)
    entry = TranslationEntry.objects.filter(obj=..., field=...).first()

    if entry is None:
        → шукаємо в Translation Memory: (kind, source_hash, target_lang, prompt_version) зі status=APPROVED
            знайшли → копіюємо target_text, status=APPROVED, source='memory', $0 витрат
            не знайшли → створюємо PENDING
    elif entry.source_hash == new_hash:
        → нічого не робимо (0 запитів, 0 грошей)   ← 95% випадків при щоденному синку
    else:
        → entry.source_text = new_source; entry.source_hash = new_hash
          entry.status = STALE
          # published_text НЕ чіпаємо — сайт далі показує старий, схвалений RU
          → enqueue re-translation → status = PENDING_REVIEW
```

**Translation Memory** — це та сама таблиця, тільки запит по `(kind, source_hash)` без прив'язки до об'єкта. Тому дублікати назв/описів між товарами перекладаються один раз безкоштовно.

    input_tokens   = models.IntegerField(default=0)
    output_tokens  = models.IntegerField(default=0)
    cost_usd       = models.DecimalField(max_digits=8, decimal_places=6, default=0)

    edited_by      = models.ForeignKey(User, null=True, blank=True,
                                       on_delete=models.SET_NULL, related_name="+")
    approved_by    = models.ForeignKey(User, null=True, blank=True,
                                       on_delete=models.SET_NULL, related_name="+")
    approved_at    = models.DateTimeField(null=True, blank=True)
    created_at     = models.DateTimeField(auto_now_add=True)
    updated_at     = models.DateTimeField(auto_now=True)

    class Meta:
        constraints = [models.UniqueConstraint(
            fields=["content_type", "object_id", "field", "target_lang"],
            name="uniq_translation_target")]
        indexes = [
            models.Index(fields=["kind", "status"]),
            models.Index(fields=["kind", "source_hash", "target_lang"]),  # Translation Memory
        ]

    def approve(self, user):
        self.published_text = self.target_text
        self.published_at   = timezone.now()
        self.approved_by    = user
        self.approved_at    = timezone.now()
        self.status         = TStatus.APPROVED
        self.save()
        write_back_to_model(self)     # копіюємо у Product.name_ru / AttributeOption.value_ru
```

**Два поля `target_text` / `published_text` — це і є відповідь на «переклад НЕ йде одразу в продакшн».** Модель пише в `target_text`, редактор править `target_text`, і **тільки** дія «Схвалити» переносить його в `published_text`. Коли джерело змінилось — сайт далі віддає старий `published_text`, а новий машинний варіант чекає в черзі. Ніяких «на сайті раптом з'явився неперевірений текст».

**Денормалізація для фронту.** Після `approve()` значення копіюється у `*_ru`-колонку самої моделі (`django-modeltranslation` → `Product.name_ru`, `AttributeOption.value_ru`). API/Ninja читає тільки її → нуль джойнів на рендері. `TranslationEntry` лишається джерелом правди для черги, аудиту й витрат. Фолбек у серіалізаторі: `value_ru or value_uk`.

### 6.2. Celery-задачі

```
sync_prices/sync_specs (SYNC.md)
        ↓
collect_translation_targets      # створює/оновлює TranslationEntry, ставить STALE, б'є в Translation Memory
        ↓
submit_translation_batches       # групує PENDING за kind → Batch API (пріоритет: словник → категорії → назви → описи → SEO)
        ↓
poll_translation_batches (кожні 5 хв)
        ↓
ingest_batch_results             # валідатори → MACHINE або FAILED; пише usage/cost
        ↓

```
Нічний cron: `rebuild_glossary` (bump `glossary_version`, тільки зі status=APPROVED).

### 6.3. Адмінка (django-unfold)

**Черга схвалення** — окремий розділ, не змішаний із товарами:

- **Changelist** із фільтрами: `kind`, `status`, `engine_model`, «є `model_note`», «є `validation_errors`».
- **Порядок за замовчуванням — за пріоритетом впливу:** `attribute_name` → `unit` → `attribute_value` → `category_name` → `product_name` → `seo_*` → `product_description`. Логіка: схвалення 600 назв характеристик відразу «фіксує» термінологію в 10 000 товарах і в глосарії для описів.
- **Дві колонки поруч:** `UK (джерело)` | `RU (редаговане поле)`. Для HTML — рендер-прев'ю та підсвічений diff тегів.
- Значок ⚠ біля рядків із `model_note` (Claude сама позначила, де сумнівалась) — це найкращий сигнал, куди дивитись у першу чергу.
- **Bulk-дії:**
  - `Схвалити й опублікувати` (масово) — **дозволена тільки якщо `validation_errors == []`**; для `product_description` вимагає окремого прапорця «я переглянув».
  - `Відхилити` → діалог із причиною → причина йде в `user`-повідомлення наступного перекладу цього рядка (`"Попередній варіант відхилено: {reason}. Врахуй це."`) → Opus 4.8.
  - `Перекласти заново`
  - `Не перекладати` (для брендів/кодів, що просочилися у словник)
- **Інлайн-панель «Переклад»** на сторінці Product / Category / Attribute — редактор бачить RU прямо там, де редагує товар; збереження там = `edited_by=user`, `status=APPROVED`.
- **Дашборд-віджет:** `pending`, `stale`, `failed`, `витрати за поточний місяць` (сума `cost_usd`).

### 6.4. Політика схвалення — чесна пропозиція

Замовник просив ручне схвалення. Порахуймо, що це означає:

| kind | Одиниць | ~сек/од. | Годин |
|---|---|---|---|
| Словник (attr + value + unit + category) | ~4 900 | 5 | **~7 год** |
| Назви товарів | 10 000 | 5 | ~14 год |
| SEO | 20 000 | 3 | ~17 год |
| **Описи (HTML)** | 6 000 | 60 | **~100 год** |

100% ручного схвалення описів = **~2,5 робочих тижні одного модератора**. Пропозиція:

- **`attribute_*`, `unit`, `category_name` — 100% ручне схвалення.** Це 7 годин і найвищий ROI: воно фіксує термінологію для всього іншого через глосарій.
- **`product_name` — масове схвалення після автовалідації** (бренд/модель перевірені кодом), вибірковий контроль 10%.
- **`seo_*`, `product_description` — авто-публікація за умови, що всі валідатори пройшли І немає `model_note`**; усе з `model_note` або з помилками валідації → у ручну чергу (реально це буде 5–10% від обсягу, тобто ~10 год замість 100).

Політика — це `settings.TRANSLATION_AUTO_APPROVE = {kind: bool}`, тому замовник може будь-коли перемкнути на 100% ручне. Але цифри вище варто йому показати **до** старту, а не після.

Кешування (множники до base input, **стакаються з Batch**): write 5m = ×1,25 · write 1h = ×2,0 · **read = ×0,1**.

### 7.2. Вибір

**Основна модель — `claude-sonnet-5`.**
- Переклад — це не задача на ризонінг. Sonnet 5 має якість, близьку до Opus, на мовних задачах.
- 1M контекст (не потрібен, але дає запас на довгі описи).
- Найкраще співвідношення ціна/якість; на весь каталог різниця з Opus — $50 vs $130.

**`claude-opus-4-8` — тільки на двох ділянках:**
1. **Словник характеристик.** Обсяг мізерний (~4 300 коротких рядків, $0,65), а вплив максимальний: цей текст потім реплікується в 300 000 місцях каталогу і потрапляє в глосарій. Тут економити на моделі — безглуздо.
2. **Ретраї після FAILED і після відхилення модератором.**

**`claude-haiku-4-5` — НЕ використовуємо.** Економія на всьому проєкті ~$20; при цьому 200K контекст, слабша багатомовність і той самий мінімальний кешований префікс 4 096 токенів, що й у Opus 4.8 (тобто жодної переваги по кешу він не дає). Не той компроміс.
> Примітка: у первинній чернетці аргумент був «у Haiku поріг 4 096, а в Sonnet 5 — 1 024». Це неправильно: у Opus 4.8 теж 4 096, а для Sonnet 5 значення офіційно не підтверджене. Аргумент проти Haiku тримається на якості й контексті, а не на кеші.

**`claude-fable-5` — НЕ використовуємо.** Вдвічі дорожча за Opus, її сила — довгі агентні задачі, а не переклад.

### 7.3. ⚠️ Дві не очевидні речі, що подвоюють рахунок, якщо їх не зробити

1. **На Sonnet 5 adaptive thinking увімкнене за замовчуванням, коли параметр `thinking` не переданий.** (Підтверджено докою: для Sonnet 5 пропуск поля `thinking` = adaptive; для Opus 4.8/4.7 — навпаки, без thinking.) Для перекладу це чистий спалений бюджет на output-токени. **Обов'язково:**
   ```python
   thinking={"type": "disabled"},
   output_config={"effort": "low", "format": {...}},
   ```
   (`{"type": "disabled"}` приймається і на Sonnet 5, і на Opus 4.8. На Fable 5 — навпаки, 400; але ми Fable не використовуємо.)
2. **Prefill (останнє assistant-повідомлення) на всіх поточних моделях повертає 400.** Форматування JSON робимо **тільки через structured outputs** (`output_config.format` з `json_schema`), не через prefill і не через «поверни JSON, будь ласка».

### 7.4. Первинний переклад каталогу (10 000 товарів)

Припущення: 2,5 симв./токен · описи є у 60% товарів, середній ≈ 3 000 симв. HTML ≈ 1 200 токенів · system+глосарій ≈ 1 500 токенів на запит (кешується).

> ⚠️ Зверни увагу: припущення «system+глосарій ≈ 1 500 токенів» **несумісне з мінімальним кешованим префіксом 4 096 у Opus 4.8** (§5.1, §7.6). Для ділянок на Opus 4.8 (словник, категорії, ретраї) system-блок треба піднімати ≥ 4 096 токенів, інакше рядок «Читання кешу» для них буде нулем, а input рахуватиметься за повною ціною. На загальну оцінку це впливає слабко (словник+категорії = $0,95 із $52), але в коді це треба врахувати.

| Блок | Одиниць | Input | Output | Модель | Вартість (Batch) |
|---|---:|---:|---:|---|---:|
| Словник (attr + value + unit) | 4 300 | 0,043 M | 0,043 M | Opus 4.8 | **$0,65** |
| Категорії + їх SEO | 300 | 0,02 M | 0,02 M | Opus 4.8 | **$0,30** |
| Назви товарів (описова частина) | 10 000 | 0,35 M | 0,35 M | Sonnet 5 | **$2,10** |
| **Описи (HTML)** | 6 000 | 7,2 M | 7,2 M | Sonnet 5 | **$43,20** |
| SEO title + description товарів | 20 000 | 0,70 M | 0,70 M | Sonnet 5 | **$4,20** |
| Статичні сторінки + новини | ~60 | 0,12 M | 0,12 M | Sonnet 5 | **$0,72** |
| Читання кешу (system+глосарій) | ~7 000 запитів | 10,5 M (cache read) | — | — | **$1,05** |
| **РАЗОМ** | | **~18,9 M** | **~8,4 M** | | **≈ $52** |

- Без Batch API (синхронно) → **≈ $105**.
- Усе на Opus 4.8 + Batch → **≈ $130**.
- Після 01.09.2026 (кінець інтро-ціни Sonnet 5) → **≈ $78**.
- Плюс ~20% на перепереклади після відхилень модератором.

> **Бюджет на первинний прогін: $100–150.** Це для 10 000 товарів. Оцінка чутлива до одного параметра — **середньої довжини опису**: описи це 83% рахунку. Перед першим повним прогоном обов'язково:
> ```python

>                              messages=[{"role":"user","content": sample_payload}])
> ```
> на 50 реальних описах — і перерахувати. Далі: прогнати 200 товарів, порахувати фактичний `usage`, екстраполювати, і тільки тоді запускати каталог.

### 7.6. Що саме було виправлено після звірки (для протоколу)

| Твердження в первинній чернетці | Що каже офіційна дока | Наслідок для коду |
|---|---|---|
| Min cache prefix Opus 4.8 = **1 024** | **4 096** | System-блок < 4 096 tok → кеш на Opus 4.8 **мовчки не запишеться** (без помилки, просто `cache_creation_input_tokens = 0`). Треба або нарощувати глосарій, або не розраховувати на кеш у словниковому прогоні. |
| Min cache prefix Fable 5 = **512** | **2 048** | Не критично — Fable не використовуємо. |
| Min cache prefix Haiku 4.5 = **4 096** | **4 096** ✅ | Правильно, але аргумент «у Sonnet 5 краще (1 024)» — хибний, бо для Sonnet 5 значення в таблиці **не підтверджене**, а в Opus 4.8 воно таке саме (4 096). |
| Min cache prefix Sonnet 5 = **1 024** | у звіреній таблиці рядка для Sonnet 5 **немає** | Не приймати 1 024 як факт. Перед релізом заміряти емпірично: зробити два однакові запити з `cache_control` і подивитись `usage.cache_read_input_tokens` на другому. Консервативно проєктувати system-блок на ≥ 4 096 tok — тоді поріг перекрито для будь-якої з моделей. |

**Підтверджено без змін:** ціни всіх чотирьох моделей, Batch −50% (стакається з кешем), множники кешу (write 5m ×1,25 / write 1h ×2,0 / read ×0,1), adaptive thinking за замовчуванням на Sonnet 5 при пропущеному `thinking`, `thinking={"type":"disabled"}` приймається на Sonnet 5 і Opus 4.8, prefill → 400, structured outputs через `output_config.format` (`json_schema`), рівні `effort` `low|medium|high|xhigh|max` на Sonnet 5, токенізатор Sonnet 5 ≈ +30% токенів.

4.4 Не змінюй пунктуацію, регістр першої літери та порядок частин, якщо цього не вимагає
    граматика російської. Скорочення розгортай так само, як в оригіналі: якщо в оригіналі
    «Гарантійний термін» — це «Гарантийный срок», а не «Гарантия».
4.5 Категорії техніки — усталені російські назви:
